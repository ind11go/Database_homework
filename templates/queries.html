安全性说明

1. 如何防止普通用户删除数据

  • 身份认证与授权：实施基于角色的访问控制（RBAC），仅管理员角色拥有 DELETE 权限。
  • 数据库层面：通过 GRANT/REVOKE 限制数据库用户的操作权限，应用程序连接数据库时使用只读或受限账户。
  • 外键约束与触发器：设置 ON DELETE RESTRICT 或 CASCADE 策略，防止误删关联数据。
  • 操作审计：记录所有 DELETE 操作日志，便于追溯。

2. 如何限制用户只能查询数据

  • 数据库权限控制：为普通用户创建仅具有 SELECT 权限的数据库账号。
  • 应用层过滤：后端 API 对非管理员请求只路由到查询接口，拒绝 INSERT/UPDATE/DELETE 请求。
  • 视图隔离：向普通用户暴露视图而非基表，限制其可见的数据范围。

并发与恢复说明

1. 两个用户同时购买同一商品会出现什么问题

可能出现超卖问题（Overselling）：两个事务同时读取到商品 status=0，都执行购买操作，导致同一商品被卖出两次，违反"每个商品最多只能被交易一次"的约束。

2. 如何解决

  • 数据库事务 + 行级锁：在购买操作中使用 BEGIN TRANSACTION，配合 SELECT ... FOR UPDATE 对商品记录加排他锁，确保同一时间只有一个事务能修改该商品。
  • 乐观锁：在 Item 表增加 version 字段，更新时检查版本号，若已变更则回滚重试。
  • 唯一约束：Orders 表的 item_id 设置 UNIQUE 约束，作为最后一道防线。

3. 如果系统崩溃，如何恢复订单数据

  • WAL 模式（Write-Ahead Logging）：SQLite 启用 WAL 模式，事务日志先于数据写入磁盘，崩溃后可从日志恢复。
  • 定期备份：定时将 .db 文件备份到远程存储。
  • 事务原子性：购买操作封装在事务中，崩溃时未提交的事务自动回滚，不会留下不完整的状态（不会只插入订单而未更新商品状态）。
