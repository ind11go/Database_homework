## 二、从运行代码到获得最终网址的具体步骤

### 2.1 技术选型

本项目采用以下技术栈：
- **后端框架**：Flask（Python Web 框架）
- **数据库**：SQLite3（嵌入式关系型数据库）
- **前端**：HTML + CSS（Jinja2 模板引擎）
- **部署平台**：Render（免费云托管平台）
- **版本控制**：Git + GitHub

### 2.2 本地运行步骤

1. 克隆代码仓库
   ```bash
   git clone https://github.com/ind11go/Database_homework.git
   cd Database_homework
   ```

2. 安装 Python 依赖
   ```bash
   pip install -r requirements.txt
   ```

3. 初始化数据库（自动生成 campus_trade.db 并插入初始数据）
   ```bash
   python3 database.py
   ```

4. 启动本地服务器
   ```bash
   python3 app.py
   ```

5. 在浏览器中访问 `http://127.0.0.1:5000`

### 2.3 部署到 Render 的步骤

1. 将代码推送到 GitHub 公开仓库
2. 注册/登录 Render（https://dashboard.render.com/）
3. 创建 New Web Service，选择 GitHub 仓库
4. 配置参数：
   - Runtime: Python 3
   - Build Command: `pip install -r requirements.txt`
   - Start Command: `gunicorn app:app`
   - Region: Singapore
   - Plan: Free
5. 点击 Create Web Service，等待自动构建和部署
6. 部署成功后获得公网访问地址

### （一）数据库定义

本项目创建了三个数据表，并设置了完整的完整性约束：

| 表名 | 主键 | 外键 | 约束 |
|------|------|------|------|
| User | user_id | - | user_id, user_name, phone 均非空 |
| Item | item_id | seller_id → User(user_id) | price >= 0, status IN (0,1), 非空约束 |
| Orders | order_id | item_id → Item(item_id), buyer_id → User(user_id) | item_id 唯一 |

额外通过 TRIGGER 保证：orders 表插入/更新时，对应 item 的 status 必须为 1。

## 五、简答

### （八）安全性

#### 1. 如何防止普通用户删除数据

- **身份认证与授权**：实施基于角色的访问控制（RBAC），仅管理员角色拥有 DELETE 权限。普通用户登录后只能访问查询和数据展示页面，删除操作按钮对非管理员隐藏。
- **数据库层面**：通过 GRANT/REVOKE 限制数据库用户的操作权限。例如为应用创建专用数据库账号，仅授予必要的权限（如普通用户账号只给 SELECT），避免使用具有 DROP/DELETE 权限的 root 账号连接数据库。
- **外键约束与触发器**：设置 ON DELETE RESTRICT 策略，防止误删存在关联订单的商品记录。同时通过触发器在删除前检查业务规则。
- **操作审计**：记录所有 DELETE 操作日志（操作人、时间、被删数据），便于事后追溯和问责。

#### 2. 如何限制用户只能查询数据

- **数据库权限控制**：为普通用户创建仅具有 SELECT 权限的数据库账号，从数据库层面禁止任何修改操作。
- **应用层过滤**：后端 API 对非管理员请求只路由到查询接口（GET），拒绝所有 INSERT/UPDATE/DELETE 请求（POST/PUT/DELETE），返回 403 权限不足错误。
- **视图隔离**：向普通用户暴露视图（如 SoldItemsView、UnsoldItemsView）而非基表，限制其可见的数据范围和字段，即使 SQL 注入也只能访问视图允许的数据。
- **ORM/参数化查询**：使用参数化查询防止 SQL 注入，避免用户通过构造恶意输入绕过权限限制。

### （九）并发与恢复

#### 1. 两个用户同时购买同一商品会出现什么问题

可能出现**超卖问题（Overselling）**。

具体场景：
- 用户 A 和用户 B 同时浏览商品 X（status = 0）
- 两个事务 T1 和 T2 同时读取到 status = 0
- T1 执行购买：更新 status = 1，插入订单
- T2 也执行购买：同样更新 status = 1，插入订单
- 结果：同一商品被卖出两次，违反"每个商品最多只能被交易一次"的业务规则

#### 2. 如何解决

**方案一：数据库事务 + 行级锁（推荐）**

```sql
BEGIN TRANSACTION;
SELECT status FROM Item WHERE item_id = 'i003' FOR UPDATE;
-- 检查 status == 0
UPDATE Item SET status = 1 WHERE item_id = 'i003';
INSERT INTO Orders VALUES ('o003', 'i003', 'u003', '2024-05-05');
COMMIT;
```

`FOR UPDATE` 对商品记录加排他锁，确保同一时间只有一个事务能修改该商品，其他事务阻塞等待。

**方案二：乐观锁**

在 Item 表增加 version 字段：
```sql
ALTER TABLE Item ADD COLUMN version INTEGER DEFAULT 0;
```

更新时检查版本号：
```sql
UPDATE Item SET status = 1, version = version + 1
WHERE item_id = 'i003' AND version = 当前读取的版本号;
```

若版本号已变更（说明有其他事务修改），则更新失败，业务层可重试。

**方案三：唯一约束（最后一道防线）**

Orders 表的 item_id 设置 UNIQUE 约束，即使并发控制失效，数据库层面也会拒绝重复插入，保证数据一致性。

#### 3. 如果系统崩溃，如何恢复订单数据

- **WAL 模式（Write-Ahead Logging）**：SQLite 启用 WAL 模式后，所有修改先写入事务日志文件（.db-wal），再异步刷新到主数据库文件。系统崩溃后，下次打开数据库时会自动从 WAL 文件恢复未提交或已提交但未持久化的数据。
- **事务原子性**：购买操作封装在 BEGIN TRANSACTION ... COMMIT 中，崩溃时未提交的事务自动回滚。这确保了不会出现"只插入订单但未更新商品状态"的不一致中间状态。
- **定期备份**：通过定时任务（如 cron）将 .db 文件备份到远程存储（如云盘、S3），即使主数据库文件损坏也能从备份恢复。
- **主从复制（生产环境）**：对于高可用场景，可配置 SQLite 到 PostgreSQL/MySQL 的同步复制，主库故障时快速切换到从库。
